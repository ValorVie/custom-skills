# mp-prototype

原型是**回答問題的丟棄式程式碼**。問題決定形狀，答對問題比寫好程式重要。

## 先判斷分支

- 「這個邏輯／狀態模型／資料形狀對嗎？」→ 讀 [references/logic.md](references/logic.md)，
  做可互動的終端原型。
- 「這個畫面該長怎樣？」→ 讀 [references/ui.md](references/ui.md)，
  做多變體、單一路由、可切換的 UI 原型。

分支選錯整個原型就白做。問題模糊且使用者不在場時，看周邊程式碼決定
（後端模組 → logic；頁面或元件 → UI），並在原型頂端註明假設。

## 六條通則

1. **從第一天就是丟棄式，且標示清楚**：放在未來使用位置附近，
   命名讓路過的人一眼看出是原型。
2. **一鍵執行**：掛進專案既有的 task runner；沒有 task runner 就把指令寫在原型 README 頂端。
3. **預設無持久化**：狀態放記憶體。持久化是原型要**驗證**的對象，不是依賴。
4. **略過打磨**：不寫測試、不做多餘錯誤處理、不抽象。目標是快速學到東西。
5. **外顯狀態**：每次操作（logic）或每次切換變體（UI）後，完整呈現相關狀態。
6. **完成即捕捉**：見下節。

## 產物處置（本專案規則）

原型回答完問題後：

1. 驗證後的**決策**寫回真正的歸屬：實作程式碼、OpenSpec artifact、或對應工作項目。
2. 原型程式碼 commit 到 `experiment/<topic>` 分支，不進 main。
3. 在對應的 issue 或 OpenSpec change 留下分支連結與一行結論（哪個答案、為什麼）。
4. main 分支不留任何原型程式碼、變體元件或切換器。

原型是決策的一手來源，留在 experiment 分支供回查；留在 main 只會腐爛並誤導下一個讀者。
