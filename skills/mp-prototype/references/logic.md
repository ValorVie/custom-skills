# Logic 原型

一個小型互動終端 app，讓使用者親手驅動狀態模型。適用於**業務邏輯、狀態轉移、
資料形狀**這類「紙上看起來合理、實際按過才知道哪裡怪」的問題。

問題若是「該長怎樣」→ 走錯分支，改讀 [ui.md](ui.md)。

## 流程

### 1. 寫下問題

動手前，在原型 README 或檔案頂端註解寫下：要驗證的狀態模型是什麼、要回答的問題是什麼。
一段就好。答錯問題的原型是純浪費，問題外顯才能事後核對。

### 2. 選語言

用宿主專案的語言與既有工具鏈。不為原型引入新的套件管理器或 runtime。
專案沒有明顯 runtime（如純文件 repo）時，先問。

### 3. 把邏輯隔離成可攜模組

真正回答問題的邏輯放在小而純的介面後面，未來可以直接搬進正式程式碼。
終端介面是丟棄式的，邏輯模組不是。依問題選形狀：

- **純 reducer**：`(state, action) => state`。動作是離散事件、狀態是單一值時適用。
- **狀態機**：明確的狀態與轉移。「現在哪些動作是合法的」本身就是問題時適用。
- **一組純函式**：對純資料的轉換，沒有隱含的當前狀態時適用。
- **帶方法的模組**：邏輯真正擁有持續內部狀態時適用。

依問題選形狀，不是依「哪個好接終端介面」選。保持純粹：邏輯裡不做 I/O、
不碰終端、不用 console.log 控制流程。介面呼叫邏輯，方向不可逆。

### 4. 做最小的終端介面

每次操作後清畫面、重繪整個畫面（不是往下append）。一個畫面兩個部分：

1. **當前狀態**：一欄一行或格式化 JSON，欄名粗體、次要資訊淡化（ANSI escape 即可）。
2. **快捷鍵清單**：列在底部，例如 `[a] 新增 [d] 刪除 [t] 時間前進 [q] 離開`。

整個畫面塞得進一屏。迴圈：初始化狀態 → 讀一鍵 → 派發處理 → 重繪 → 直到離開。

### 5. 一鍵可跑

掛進專案 task runner（package.json scripts、Makefile、pyproject.toml）。
使用者不需要記路徑。

### 6. 交給使用者駕駛

給出執行指令。有價值的時刻是使用者說「等等，這不該發生」——那是**想法**裡的 bug，
正是原型存在的目的。要加動作就加，原型會演化。

### 7. 捕捉

驗證過的 reducer／狀態機／函式組可直接搬進正式模組；終端外殼跟著原型
進 `experiment/<topic>` 分支。其餘依 [SKILL.md](../SKILL.md) 產物處置。

## 反模式

- **不寫測試**：需要測試的原型已經不是原型。
- **不接真資料庫**：除非問題就是持久化，否則用記憶體。
- **不泛化**：沒有「以後想支援 X 怎麼辦」。一個原型回答一個問題。
- **邏輯與介面不可糊在一起**：reducer 裡出現終端碼就不可攜了。
- **外殼不進正式碼**：值得留的是後面那個純模組。
