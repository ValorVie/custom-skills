# UI 原型

在單一路由上產生**數個結構完全不同的 UI 變體**，用浮動底欄切換。使用者在瀏覽器裡
翻看、挑一個（或各取所長），其餘丟棄。

問題若是邏輯或狀態 → 走錯分支，改讀 [logic.md](logic.md)。

## 兩種掛法（強烈偏好 A）

變體**貼著真實 app 環境**（真 header、真 sidebar、真資料密度）才好評判；
孤立的空路由是真空，每個變體看起來都不錯。

### 掛法 A：掛在既有頁面（預設）

路由已存在。變體渲染在**同一路由**，用 `?variant=` 參數切換。既有的資料抓取、
參數、權限全部保留，只換渲染子樹。要原型的東西還沒有頁面、但自然屬於某頁的一部分
（dashboard 的新區塊、settings 的新卡片）也算掛法 A：把變體掛進宿主頁。

### 掛法 B：新開頁面（最後手段）

僅在真的沒有任何既有頁面可寄宿時使用（全新的頂層介面）。依專案既有路由慣例
開一條丟棄式路由，路徑或檔名帶 `prototype` 字樣，同樣用 `?variant=`。

## 流程

### 1. 寫下問題、定變體數

預設 **3 個變體**，上限 5（再多就不是差異是雜訊）。計畫一行寫在原型位置：
「settings 頁三個變體，`?variant=` 切換，掛在既有 `/settings` 路由」。

### 2. 產生結構性不同的變體

每個變體遵守：頁面目的與可用資料、專案既有的元件庫與樣式系統、
清楚的匯出名（`VariantA`、`VariantB`、`VariantC`）。

變體必須**結構性不同**——不同版面、不同資訊層級、不同主要操作，
不是換色換文案。兩個草稿太像就重做一個，明確給「不准用卡片格」這類反向約束。

### 3. 接上切換器

路由上一個 switcher 元件依 `?variant=` 渲染對應變體。掛法 A 的資料抓取
全部留在 switcher 之上，每個變體只換渲染子樹。

### 4. 浮動底欄

畫面底部置中的小工具列：左箭頭、變體標籤（鍵 + 名稱，如 `B — Sidebar layout`）、
右箭頭，循環切換。行為：

- 點擊用框架 router 更新 `?variant=`，可分享、重整後不變。
- `←` `→` 鍵盤也能切；焦點在輸入框時不攔截。
- 視覺上明顯不屬於被評估的設計（高對比 pill）。
- 用 `NODE_ENV !== 'production'` 之類的檢查隱藏，誤合併也不會出現在正式環境。

### 5. 交給使用者翻看

給出網址與 `?variant=` 鍵。最有價值的回饋通常是「我要 B 的 header 配 C 的 sidebar」
——那才是使用者真正要的設計。

### 6. 捕捉與清理

勝出變體定案後：勝者摺回正式程式碼（原型約束下寫的碼要重寫乾淨，不直接晉升）；
全套變體與切換器進 `experiment/<topic>` 分支；main 不留變體與切換器。
其餘依 [SKILL.md](../SKILL.md) 產物處置。

## 反模式

- **只差顏色或文案的變體**：那是微調不是原型。真變體對結構有歧見。
- **變體間共用太多**：共用 `<Header>` 可以，共用 `<Layout>` 就失去意義。
- **變體接真 mutation**：唯讀即可，要改就打 stub。問題是「長怎樣」不是「後端通不通」。
- **原型直接晉升正式碼**：原型約束（無測試、極簡錯誤處理）寫出的碼，摺入時要重寫。
