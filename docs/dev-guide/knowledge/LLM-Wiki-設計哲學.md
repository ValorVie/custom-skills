# LLM Wiki 設計哲學

## 概述

LLM Wiki 是 Andrej Karpathy 提出的知識管理模式：讓 LLM 增量建構與維護一個持久的 wiki，取代傳統的 RAG（檢索增強生成）方式。這不是一個特定工具，而是一套設計哲學，描述如何讓 AI 成為知識庫的維護者。

本專案中的 [Personal Wiki Skill](Personal-Wiki-Skill-個人知識編譯.md) 和 [Graphify](Graphify-知識圖譜建構.md) 都是基於這套哲學衍生出的具體實作。

## 核心思想

### RAG 的侷限

傳統 RAG 流程：上傳文件集 → LLM 在查詢時檢索相關片段 → 生成回答。問題在於 LLM 每次都從零開始重新發現知識，沒有累積。需要綜合五份文件才能回答的問題，每次都要重新找出並拼湊相關片段。NotebookLM、ChatGPT 檔案上傳、大多數 RAG 系統都是這個模式。

### 持久編譯的 Wiki

LLM Wiki 的做法不同：LLM **增量建構並維護一個持久的 wiki** — 一組結構化、互相連結的 markdown 檔案，位於使用者與原始來源之間。

當加入新來源時，LLM 不只是為後續檢索建立索引。它會閱讀內容、提取關鍵資訊、整合進既有的 wiki — 更新實體頁面、修訂主題摘要、標記新資料與舊主張的矛盾、強化或挑戰持續演進的綜合分析。

> 知識被編譯一次，然後持續維護，而非每次查詢時重新推導。

### 人機分工

- **人類**：策展來源、探索方向、問對的問題
- **LLM**：摘要、交叉引用、歸檔、記帳 — 所有讓知識庫長期有用的繁瑣工作

實際操作：一邊開著 LLM agent，另一邊開著 Obsidian。LLM 根據對話進行編輯，使用者即時瀏覽結果 — 跟隨連結、查看圖形視圖、閱讀更新的頁面。

**類比：Obsidian 是 IDE，LLM 是工程師，wiki 是程式碼庫。**

## 三層架構

| 層級 | 角色 | 說明 |
|------|------|------|
| 原始來源 (Raw Sources) | 不可變的事實來源 | 文章、論文、圖片、資料檔。LLM 只讀取不修改 |
| Wiki | LLM 擁有的知識層 | 摘要、實體頁、概念頁、比較、綜述。LLM 建立、更新、維護交叉引用 |
| 模式定義 (Schema) | 行為規範 | CLAUDE.md 或 AGENTS.md，定義 wiki 結構、慣例、工作流程。人與 LLM 共同演進 |

## 核心操作

### 匯入 (Ingest)

將新來源放入原始集合，指示 LLM 處理。流程範例：

1. LLM 閱讀來源
2. 與使用者討論重要發現
3. 在 wiki 中撰寫摘要頁
4. 更新索引
5. 跨 wiki 更新相關的實體與概念頁
6. 在日誌中新增記錄

一個來源可能觸及 10-15 個 wiki 頁面。可以逐一匯入（保持參與）或批量匯入（較少監督）。

### 查詢 (Query)

對 wiki 提問。LLM 搜尋相關頁面、閱讀、綜合帶引用的回答。回答可以是 markdown 頁面、比較表、投影片 (Marp)、圖表 (matplotlib)、畫布。

**好的回答可以歸檔回 wiki 成為新頁面。** 這樣探索成果也會在知識庫中累積，就像匯入的來源一樣。

### 維護 (Lint)

定期請 LLM 健康檢查 wiki：

- 頁面間的矛盾
- 新來源已取代的過時主張
- 沒有入站連結的孤立頁面
- 提及但缺少獨立頁面的重要概念
- 缺失的交叉引用
- 可透過搜尋填補的資料缺口

## 索引與日誌

| 檔案 | 用途 | 特性 |
|------|------|------|
| `index.md` | 內容導向的目錄 | 每頁列出連結、摘要、metadata。按分類組織。LLM 每次匯入時更新，查詢時先讀索引再深入 |
| `log.md` | 時序記錄 | 純附加的事件紀錄：匯入、查詢、維護。統一前綴格式便於 unix 工具解析 |

在中等規模（約 100 個來源、數百頁）下，索引檔足以取代嵌入式 RAG 基礎設施。

## 適用場景

| 場景 | 說明 |
|------|------|
| 個人知識 | 追蹤目標、健康、心理、自我提升。從日記、文章、筆記中建構結構化自我畫像 |
| 研究 | 持續數週至數月的深度研究。閱讀論文、報告，增量建構帶演進論點的 wiki |
| 閱讀 | 逐章歸檔，建構角色、主題、情節線索頁面。讀完後擁有豐富的伴讀 wiki |
| 團隊/企業 | 從 Slack、會議記錄、專案文件、客戶電話中維護內部 wiki。LLM 做沒人想做的維護工作 |
| 競品分析 | 持續累積知識的任何領域：盡職調查、旅行規劃、課程筆記、興趣深潛 |

## 為什麼有效

維護知識庫的繁瑣部分不是閱讀或思考，而是記帳：更新交叉引用、保持摘要更新、標記新舊資料矛盾、跨頁面維護一致性。人類會因為維護負擔超過價值而放棄 wiki。LLM 不會無聊、不會忘記更新交叉引用，且能一次修改 15 個檔案。wiki 能持續維護，因為維護成本趨近於零。

這個想法在精神上與 Vannevar Bush 1945 年提出的 Memex 相關 — 一個帶有文件間關聯軌跡的個人策展知識庫。Bush 的願景比後來的 Web 更接近這個概念：私有、主動策展、文件間的連結與文件本身同樣有價值。他無法解決的問題是誰來做維護。LLM 解決了這個問題。

## 工具建議

| 工具 | 用途 |
|------|------|
| Obsidian | wiki 瀏覽器。圖形視圖觀察 wiki 形態，Dataview 外掛查詢 frontmatter |
| Obsidian Web Clipper | 瀏覽器擴充套件，將網頁文章轉為 markdown |
| [qmd](https://github.com/tobi/qmd) | 本地 markdown 搜尋引擎，支援 BM25/向量混合搜尋與 LLM 重排。規模擴大後替代索引檔 |
| Marp | markdown 投影片格式，直接從 wiki 內容生成簡報 |
| Git | wiki 就是 markdown 檔案的 git repo，免費獲得版本歷史、分支、協作 |

## 與其他工具的關係

LLM Wiki 是設計哲學，不是具體實作。以下兩個工具分別在不同領域實現了這套哲學：

| 工具 | 定位 | 輸入 | 輸出 | 核心差異 |
|------|------|------|------|----------|
| [Personal Wiki Skill](Personal-Wiki-Skill-個人知識編譯.md) | 個人資料編譯器 | 日記、筆記、訊息等個人資料 | 敘事性 wiki 文章 (markdown + wikilinks) | 強調理解與綜合，產出是「心智地圖」 |
| [Graphify](Graphify-知識圖譜建構.md) | 程式碼/文件圖譜建構器 | 程式碼、文件、論文、圖片 | 知識圖譜 (NetworkX + HTML 視覺化) | 強調結構與關係，產出是「架構地圖」 |

三者的層級關係：

```
LLM Wiki 設計哲學（本文）
├── Personal Wiki Skill — 個人知識領域的實作
└── Graphify — 程式碼/文件領域的實作
```
